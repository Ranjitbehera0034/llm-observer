1. Executive Summary
LLM Observer is a privacy-first, local-only developer tool that tracks AI API costs, usage, and performance across multiple LLM providers. It serves both individual developers and engineering teams — running entirely on each developer's machine while offering an optional encrypted team dashboard for company-wide visibility.
The product addresses a massive and growing pain point: as developers and businesses use OpenAI, Claude, Gemini, and other LLMs simultaneously, tracking token usage, managing costs, and preventing runaway API bills becomes a nightmare. A single infinite loop bug can rack up thousands of dollars in API charges overnight.

Key Differentiator
Unlike cloud-based competitors (Helicone, Portkey, LangSmith) that require routing API traffic through their servers, LLM Observer runs a lightweight local proxy. This eliminates the trust barrier — developers never need to hand over their API keys or route sensitive prompt data through a third-party server. For teams, only anonymized aggregated cost data is synced to the cloud — never prompts, responses, or API keys.

Dual-Audience Strategy
Individual developers are the acquisition engine — they discover the tool, adopt it for free, and spread word-of-mouth. Teams are the revenue engine — they pay higher prices, are stickier, and have longer retention. The individual product IS the team product; each team member runs the same local tool, with the team layer adding shared visibility on top.

4. System Architecture
The architecture is built around a single core principle: nothing leaves the developer's machine. All processing happens locally — the proxy intercepts LLM API calls, logs them to a local SQLite database, and serves a dashboard from localhost.

Architecture Flow
1. Developer's app sends LLM request to localhost:4000 (instead of directly to OpenAI/Claude/Gemini)
2. Local proxy intercepts the request, authenticates the project key, and checks budget limits
3. If budget is OK, proxy forwards the request directly to the LLM provider's API
4. Provider responds — proxy captures the response, counts tokens, calculates cost
5. Proxy logs everything to local SQLite asynchronously (non-blocking — no latency added)
6. Response is returned to the developer's app unchanged — the app never knows the proxy exists
7. Dashboard reads from SQLite and displays real-time stats via Server-Sent Events (SSE)

Budget Kill Switch Flow
The kill switch is the single most valuable feature — it prevents runaway API costs from bugs, infinite loops, or misconfigured retry logic.
	•	Request arrives at proxy
	•	Budget check: today's spend ($47.20) vs daily limit ($50.00)
	•	LIMIT EXCEEDED: $47.20 >= $50.00 → BLOCK
	•	Return HTTP 429 with clear error: 'LLM Observer: Daily budget limit ($50.00) reached. Requests blocked until midnight UTC.'
	•	Send native OS notification + optional Slack/Discord webhook
	•	Log blocked request with status='blocked_budget' for auditing

Dashboard Real-Time Updates
The dashboard receives real-time updates via Server-Sent Events (SSE), which is a lightweight persistent HTTP connection. This avoids the overhead of WebSocket while providing instant updates whenever a new LLM request is logged.
Flow: New request logged → SSE event emitted → React state updates → Charts, counters, and log table refresh instantly. No polling required.

7. API Surface — Proxy Endpoints
These are the endpoints that the developer's application calls. They are drop-in replacements for the original provider APIs — same request format, same response format. The only change is the base URL.

Integration Example
// BEFORE (direct to OpenAI)
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

// AFTER (through LLM Observer — ONE line added)
const openai = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
  baseURL: "http://localhost:4000/v1/openai",  // only change

11. Tauri Desktop Migration Plan

Why Tauri (Not Electron)
LLM Observer is a lightweight utility tool that developers keep running 8+ hours a day in the background. Electron bundles an entire Chromium browser (200MB, 100-300MB RAM). Tauri uses the native OS webview (5-10MB, 20-50MB RAM). For an always-on background utility, this difference is critical to user adoption and satisfaction.

Sidecar Strategy (Phase 2)
The sidecar approach means your battle-tested Node.js proxy code runs exactly as-is inside Tauri. No rewriting, no new bugs. You get Tauri benefits immediately while learning Rust gradually.

	•	Install Rust + Tauri CLI: cargo install tauri-cli
	•	Initialize Tauri in apps/tauri/: cargo tauri init
	•	Point Tauri's webview to packages/dashboard build output
	•	Bundle Node.js proxy as sidecar binary using pkg or nexe
	•	Tauri launches sidecar on app start — proxy runs in background
	•	Add system tray icon with status indicator
	•	Add native OS notifications via tauri-plugin-notification
	•	Configure auto-start on login via tauri-plugin-autostart
